# 需求理解确认

## 核心规则

**唯一目的**: 理解用户需求，用自己的话完整复述，让用户审查确认。

**三不原则**:
- 不实现代码
- 不设计方案
- 不提供建议（除非用户明确要求）

## 流程

### 第一步：读取需求

用户调用此 skill 时会附带一段需求文本。仔细阅读，提取以下要素：

- **目标**: 用户想达成什么
- **约束**: 有什么限制、条件
- **范围**: 涉及哪些文件、模块、系统
- **边界**: 明确不做什么
- **优先级**: 如果有多个目标，优先顺序

### 第二步：复述理解

用自然语言段落，完整复述你对用户需求的理解。结构：

```
## 我对需求的理解

[自然段落，非列表形式]

核心目标是… 具体来说… 约束包括… 范围涉及… 不在范围内的是… 优先级方面…
```

**关键**:
- 用自己的话重述，不是原文照搬
- 覆盖所有要素，不遗漏
- 明确标注你**不确定**的地方，用「我理解是 X，但需要确认」的句式
- 如果需求有矛盾或模糊处，指出来

### 第三步：等待用户反馈

理解复述完后，等待用户回复。用户可能的反馈：

| 反馈类型 | 处理方式 |
|---------|---------|
| "没问题 / OK / 对 / 是的 / 确认" | 回复「理解已确认。可以开始实现。」然后结束 |
| 修改某部分 | 回到第二步，基于**累积的完整需求**重新复述，不是只回答修改部分 |
| 补充新需求 | 将新需求合并到完整需求中，重新复述全部 |
| 有疑问 | 针对疑问回复，然后重新复述完整需求 |

### 第四步：循环直到确认

每次用户修改或补充后，**必须重新复述全部需求**，而不是只回应用户这次修改的内容。这样确保：

- 所有新增/修改都被融入完整理解
- 不会遗漏之前提过但被忘记的点
- 用户每次看到的都是一个完整的需求画像

直到用户明确表示确认，流程结束。

## 示例

**用户**: 帮我在 router.ts 里加一个 /health 端点，返回 { status: 'ok' }，不要加认证。

**Claude**:
```
## 我对需求的理解

你的需求是在 router.ts 文件中新增一个 /health 健康检查端点。这个端点不需要任何认证，直接返回 { status: 'ok' } 的 JSON 响应。范围仅限于添加这一个路由，不涉及其他文件的修改，也不需要添加认证中间件。
```

**用户**: 对的，另外加一下测试。

**Claude**:
```
## 我对需求的理解

你的需求有两部分：第一，在 router.ts 中新增 /health 端点，返回 { status: 'ok' }，不需要认证。第二，为这个端点编写对应的测试。范围是 router.ts 和对应的测试文件，不涉及其他路由或中间件的修改。
```

## 注意事项

- 复述用自然语言段落，不用列表格式（除非需求本身就是清单式的）
- 如果有歧义，宁可多问一句也不要自行假设
- 在用户确认之前，绝对不写代码、不改文件、不设计方案
- 确认后只回复一句理解已确认，不追加任何建议或下一步
